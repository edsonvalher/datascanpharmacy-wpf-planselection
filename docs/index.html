In what order do we migrate
the ~400 COBOL screens?

We cannot migrate them all at once. This decision defines the sequence and pace of the entire project.

Affects Delivery speed · Visible ROI · Technical risk
The concrete question: Do we start with the easiest screens (fewest dependencies), the most business-critical ones, or do we take a complete functional domain end-to-end?
A
By coupling
Easiest screens first

Choose screens with the fewest dependencies on other COBOL programs. Safest to migrate, but usually secondary features the team does not notice as real progress.

Pros
+Lower technical risk
+Team builds confidence
+Good starting point

−No visible real progress in the system
−Visible ROI very late
−Hard to sustain momentum

How do COBOL and C# coexist
during migration?

The system never stops. COBOL keeps running while C# is being built. How do the two worlds connect?

Affects Rollback speed · Operational overhead · Technical debt
The concrete question: If the newly deployed C# screen fails in production, how long does recovery take? How clean is the architecture while they coexist?
⚠
ScreenIO Constraint — What makes this migration different

Winpharm screens are not plain COBOL screens. They are rendered by ScreenIO (GS Runtime Rev 9), a third-party framework by Greenhouse Software that owns the entire event loop. This changes the nature of coexistence completely.

Cannot do
✗Call a ScreenIO screen from C# — ScreenIO has no API for that
✗Mix ScreenIO and WPF in the same window — they are separate rendering systems
✗Show half a COBOL screen and half a WPF screen side by side
How it actually works
✓Coexistence happens at the process and data level, not visually
✓Migrated screens open in WPF — non-migrated screens still open in ScreenIO
✓Each migrated feature is a complete self-contained replacement, not an overlay
A
Pure Strangler Fig
★ Recomendado
C# wraps COBOL without touching it

The C# screen exists in parallel to the COBOL program. When ready and tested, it activates. COBOL goes dormant. Simple to understand, easy to roll back.

Pros
+No changes to original COBOL
+Simple rollback if it fails
+Clear mental model for the team
Cons
−Two versions in parallel
−Manual rollback, not instant
✓ StandAlonePlan already uses this pattern today
Complexity
Low
B
Feature Flag
One switch, flips in seconds

A config flag decides whether each screen uses C# or COBOL. If something fails in production, flip the flag and revert to COBOL in seconds.

Pros
+Instant rollback
+Granular control per screen
+Ideal for gradual rollouts
Cons
−Two active implementations
−Higher maintenance overhead
−Stale flags = technical debt
Complexity
Medium
C
Branch by Abstraction
★ También usado
One interface, two implementations

Define a C# interface (IPlanRepository, IPatientRepository). First it calls COBOL; later replaced by SQL. The rest of the system never knows what's underneath.

Pros
+Cleaner architecture
+Domain has no COBOL knowledge
+Standard enterprise pattern
Cons
−More upfront work
−Requires stable COBOL Interop DLL
◆ ICobolInteropService already exists in Datascan.Winpharm.Interop
Complexity
Medium

When do we migrate the data
from ISAM to SQL?

COBOL guarda todo en ISAM files (archivos indexados en disco). C# necesita SQL. Hay 33+ ISAM files cubriendo pacientes, planes, prescripciones y más.

Affects Data integrity · Operational complexity
The concrete question: Do the data stay in ISAM while C# reads via COBOL, do we move them module by module, or do we do a big-bang migration of all 33+ files at once?
A
Per feature / module
★ Recomendado
Data travels with its screen

When migrating Plan Selection, its data is also migrated (RX11FILE → plans table in SQL). Each domain carries its data. Incremental and controlled.

Pros
+No ISAM↔SQL synchronization
+Clear scope per domain
+Verifiable before going live
Cons
−Data temporarily in two places
−Migration script per module
✓ MockPlanRepository simulates RX11/RX18 today → RealPlanRepository (SQL) next
Complexity
Medium
B
Data first
All to SQL before touching the UI

Migrar los 33+ ISAM files a SQL de una sola vez antes de migrar ninguna pantalla. Las pantallas C# nacen con datos limpios desde el primer día.

Pros
+C# screens with real data
+SQL as single source of truth
Cons
−COBOL keeps writing to ISAM
−Permanent ISAM↔SQL synchronization
−Real risk of data corruption
−Extremely high operational complexity
✗ Not recommended — synchronization overhead too high
Complexity
Very high
C
Data last
COBOL owns the data until the very end

C# reads and writes through RealRepository which talks to COBOL. Only when all screens are migrated do the data move to SQL and COBOL is retired.

Pros
+Single migration event
+COBOL always source of truth
Cons
−COBOL DLL Bridge lives for years
−Accumulated technical debt
−No SQL benefits until the end
Complexity
Low (initially)

How TDD works
in this migration

In a COBOL migration there is no external documentation — the COBOL source code is the specification. TDD is how we capture that behavior before replacing it.

Testing stack xUnit · Moq · FlaUI · Contract Tests
The base rule: COBOL source code → defines expected behavior → becomes a test → C# must pass that test. If the test passes, the migration is behaviorally equivalent to COBOL.
The 3-state cycle — Red · Green · Real
❌
RED
Write the test first

Write a test that describes the behavior found in COBOL. Run it. It fails — because the C# code does not exist yet. The runner shows red.

✅
GREEN
Write the minimum code

Write the minimum C# code to make the test pass. Run it. It passes. The runner shows green. Nothing more, nothing less.

🗄
REAL

Replace the Mock (in-memory data) with RealRepository against the SQL database. Run the exact same tests. They must still be green.

Complete example — SelectPlan step by step
01
Define the contract (the interface)

Before any test or code, define WHAT the repository must do. Only the signature — no implementation.

// The contract only — nothing more public interface IPlanRepository { IEnumerable<Plan> GetPlansByPatient(string patientId); PlanSelectionResult SelectPlan(string patientId, string planId); }
02
Implement the Mock (in-memory data)

The Mock does not touch any database. It simulates behavior with in-memory dictionaries — enough to develop and test without infrastructure.

public class MockPlanRepository : IPlanRepository { private readonly Dictionary<string, List<Plan>> _plans = new() { ["PAT-001"] = new List<Plan> { new Plan { Id = "PLAN-A", Name = "Blue Cross", IsActive = true }, new Plan { Id = "PLAN-B", Name = "Old Plan", IsActive = false } } }; public PlanSelectionResult SelectPlan(string patientId, string planId) { var plan = _plans[patientId].FirstOrDefault(p => p.Id == planId); if (plan is null) return PlanSelectionResult.NotFound; if (!plan.IsActive) return PlanSelectionResult.PlanNotActive; return PlanSelectionResult.Success; } }
03
❌ Red — write the test before the code

The test describes the behavior found in COBOL. It runs and fails because SelectPlanUseCase does not exist yet.

[Fact] public void SelectPlan_WhenPlanIsInactive_ReturnsError() { // Arrange — Mock, no database needed IPlanRepository repo = new MockPlanRepository(); var useCase = new SelectPlanUseCase(repo); // Act var result = useCase.Execute(patientId: "PAT-001", planId: "PLAN-B"); // Assert — COBOL behavior: inactive plan must be rejected Assert.Equal(PlanSelectionResult.PlanNotActive, result); }
❌ FAIL — SelectPlanUseCase does not exist yet
04
✅ Green — write the minimum code to pass

Implement the UseCase with the minimum necessary code. Nothing more. The test passes.

public class SelectPlanUseCase { private readonly IPlanRepository _repository; public SelectPlanUseCase(IPlanRepository repository) => _repository = repository; public PlanSelectionResult Execute(string patientId, string planId) => _repository.SelectPlan(patientId, planId); }
✅ PASS — SelectPlan_WhenPlanIsInactive_ReturnsError
05
🗄 Real — same test, real database

Implement RealPlanRepository against SQL. The exact tests from step 03 run again. If they pass, the migration is safe.

public class RealPlanRepository : IPlanRepository { public PlanSelectionResult SelectPlan(string patientId, string planId) { using var conn = new MySqlConnection(_connectionString); var plan = conn.QueryFirstOrDefault<Plan>( "SELECT * FROM plans WHERE patient_id=@patientId AND id=@planId", new { patientId, planId }); if (plan is null) return PlanSelectionResult.NotFound; if (!plan.IsActive) return PlanSelectionResult.PlanNotActive; return PlanSelectionResult.Success; } }
✅ PASS — against real database
06
One line activates the real world

No test is modified. No logic changes. The interface is the guarantee.

// Development and testing — no database services.AddSingleton<IPlanRepository, MockPlanRepository>(); // Production — real database services.AddSingleton<IPlanRepository, RealPlanRepository>();
What Contract Tests guarantee

Contract Tests run the same test suite against both implementations. This guarantees that Mock and Real behave identically before switching in production.

→

✅ GetPlans_WhenPatientExists
✅ SelectPlan_WhenInactive

If both pass — the Mock → Real switch is safe for production.
// Abstract base — defines the contract public abstract class PlanRepositoryContractTests { protected abstract IPlanRepository CreateRepository(); [Fact] public void SelectPlan_WhenPlanIsInactive_ReturnsError() { var repo = CreateRepository(); var result = repo.SelectPlan("PAT-001", "PLAN-B"); Assert.Equal(PlanSelectionResult.PlanNotActive, result); } } // Run against Mock public class MockPlanContractTests : PlanRepositoryContractTests { protected override IPlanRepository CreateRepository() => new MockPlanRepository(); } // Run against Real — same tests public class RealPlanContractTests : PlanRepositoryContractTests { protected override IPlanRepository CreateRepository() => new RealPlanRepository(TestConnectionString); }
